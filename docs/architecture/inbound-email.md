# Inbound Email (Poll-IMAP) Architecture

The poll-IMAP email adapter is the first **pull** inbound source. It rides the
inbound-ingestion primitive (see [webhooks.md](webhooks.md) for the push side and
the shared spine): an external message becomes a normalized `InboundMessage` on a
durable spine that `after:ingest:email` functions consume — the same path the
webhook adapter uses, with a different transport at the edge.

It is opt-in behind the `inbound-email` Cargo feature.

```text
  IMAP mailbox ─poll─► fetch(uid > cursor) ─► normalize ─► spine ─► after:ingest:email
  (transport)          (UID watermark)        (pure MIME)  (emit)   (durable dispatch)
```

---

## Design constraints (banked)

- **Poll-IMAP only** — no long-lived IMAP-IDLE connection. The adapter is
  *stateless with a cursor*: each poll opens a short-lived IMAPS connection,
  fetches, and disconnects.
- **Sending stays per-user** — this adapter only *receives*. Any paired outbound
  email is a separate, per-user concern (a later phase).
- **Transport is rustls** — IMAPS (implicit TLS) only, no OpenSSL. Fetching uses
  `BODY.PEEK[]`, so polling never marks messages `\Seen`.

---

## Configuration

Each connected account is one `[mailbox.<name>]` section. The section name is the
account's stable identity — it names the cursor row, so do not rename a mailbox in
production. The poll-IMAP *receive* half lives under `[mailbox.<name>.imap]`; the
SMTP *send* half (used by the `send_email` host op) lives under
`[mailbox.<name>.smtp]`. Either half is optional — a `[mailbox.<name>.imap]`-only
account only receives; a `[mailbox.<name>.smtp]`-only account only sends.

```toml
[mailbox.support.imap]
host = "imap.example.com"        # also the TLS SNI / certificate name
port = 993                        # default 993 (IMAPS)
username = "support@example.com"
password_env = "SUPPORT_IMAP_PASSWORD"   # password is read from the environment
mailbox = "INBOX"                 # default "INBOX"
poll_interval_secs = 60           # default 60
batch_size = 50                   # max messages processed per poll; default 50
attachment_bucket = "inbound"     # optional; attachments dropped if unset

# Declared dedicated-address routing (optional). A message to a rule's address
# maps to `entity_type`; the recipient's plus-tag becomes the entity id
# (`support+ticket-42@…` → Ticket / 42).
[[mailbox.support.imap.routing]]
address = "support@example.com"
entity_type = "Ticket"
```

A mailbox whose IMAP `password_env` is unset is **skipped with a warning**, never
polled without credentials. Requires a PostgreSQL pool.

> **Renamed (breaking).** The receive config moved from `[imap.<name>]` to
> `[mailbox.<name>.imap]` so one connected account can carry both its inbound and
> outbound halves under one section name. Update any `[imap.*]` sections to
> `[mailbox.*.imap]` (and `[[imap.*.routing]]` to `[[mailbox.*.imap.routing]]`).

## The normalization layer (pure)

Everything above transport is pure, always compiled, and unit-tested — it lives
in `fraiseql-functions` (`normalize_email`) and is shared with any future
email-shaped source. From the raw RFC 5322 bytes it derives:

- **Dedup key** — the `Message-ID` (angle brackets stripped); a message with no
  `Message-ID` falls back to a `sha256:` digest of the raw bytes, so a re-fetch
  still deduplicates.
- **Thread key** — the conversation root: the first `References` entry, else
  `In-Reply-To`, else the message's own `Message-ID`. A whole reply chain collapses
  to one key.
- **Addresses / subject / bodies** — sender and recipients (bare addresses, so
  they stay routable), subject, and text + HTML bodies (legacy charsets decoded to
  UTF-8).
- **Attachments** — decoded and streamed to `[storage]` (see below).
- **Headers** — the top-level headers, lower-cased.
- **Classification** — see the taxonomy below.

### Classification taxonomy

Reply-awareness must treat a person's reply differently from automated mail.
Exactly one class is resolved per message, most-specific first:

| Class | Signal |
|-------|--------|
| `Bounce` | RFC 3464 `multipart/report; report-type=delivery-status`, a `MAILER-DAEMON` / `postmaster` sender, or an `X-Failed-Recipients` header. |
| `OutOfOffice` | `Auto-Submitted: auto-replied`, `Precedence: auto_reply`, or a vacation-responder header. |
| `Challenge` | A challenge-response header, or an automated message whose subject asks the sender to confirm themselves (deliberately conservative — a human "please confirm" is never a challenge). |
| `AutoGenerated` | `Auto-Submitted: auto-generated`, `Precedence: bulk`/`list`/`junk`, or list headers (`List-Id` / `List-Unsubscribe`). |
| `Human` | none of the above — the only class that should advance reply-awareness. |

`Classification::is_automated()` is the loop guard: an automated message must not
be acted on as if a person had replied, and must never trigger an auto-response
that could form a mail loop.

## Cursor semantics (at-least-once)

The only state kept between polls is, per mailbox, the IMAP `UIDVALIDITY` last
seen and the highest `UID` already ingested (`_fraiseql_inbound_email_cursor`).

- Each poll fetches `UID (last_uid + 1):*` under the mailbox's *current*
  `UIDVALIDITY`.
- A changed `UIDVALIDITY` means the UID space was reset, so the watermark is
  discarded and the mailbox is re-scanned from the start.
- The cursor advances only past messages that **committed** to the spine. A
  transient failure mid-batch stops there without advancing, so the next poll
  retries from exactly that point.
- Re-fetches (retry, or a `UIDVALIDITY` reset) are deduplicated on the spine by
  `Message-ID`, so at-least-once delivery never double-dispatches. A malformed
  message is skipped (the cursor advances past it) rather than wedging the mailbox.

## Attachments and raw retention

When an `attachment_bucket` and a storage backend are configured, the raw message
(`…/raw.eml`) and each attachment are streamed into storage with deterministic
keys, and the `InboundMessage` records the resulting storage refs. When neither is
configured, attachments are **dropped with a warning** and the message is still
ingested with its bodies and headers intact — reply-awareness needs the bodies,
not the attachments.

**Current limitations / follow-ups:** attachment size / MIME-type limits, a
virus-scanning hook point, and wiring the production `StorageState` backend (only
the legacy storage backend is bridged today) are tracked follow-ups.
