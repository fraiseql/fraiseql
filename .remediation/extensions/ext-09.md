## Track Z — Proc-Macro Correctness (Priority: High)

### Z1 — `#[traced]` macro holds `span.enter()` guard across `.await` — broken on multi-threaded runtime

**File:** `crates/fraiseql-observers-macros/src/lib.rs`, lines 56–84

**Observed code (generated output for async functions):**

```rust
// Generated by #[traced(name = "some_name")]
pub async fn decorated_fn(...) {
    let span = tracing::debug_span!("some_name");
    let _guard = span.enter();           // ← guard acquired HERE
    let start = std::time::Instant::now();

    let result = async {
        /* original function body with .await points inside */
    }.await;                             // ← guard is held across this .await

    // ... duration logging ...
    result
}
```

**Why this is wrong.** The `tracing` crate documentation ([§ Using `Span`
guards](https://docs.rs/tracing/latest/tracing/struct.Span.html#in-asynchronous-code)) says
explicitly:

> *"Dropping a `Span` guard across an `.await` point will cause incorrect trace data to be
> emitted. Because `async` executors may interleave the execution of multiple futures, the
> guard will be entered on the same OS thread as the future that created it, but may be dropped
> on a different thread. This can cause incorrect trace data to be emitted."*

The `_guard` is a local variable in the outer async function. When `async { fn_body }.await`
suspends (because `fn_body` contains an `.await` that suspends), the outer function suspends
too — but `_guard` remains in scope (it is dropped when the outer future resolves, not when it
suspends). On `tokio`'s multi-threaded scheduler, the resumed future may execute on a different
thread than the one that acquired the guard. This causes:

1. **Silent span corruption.** Tracing backends receive events attributed to a span that was
   entered on thread A and exited on thread B — an impossible state. Jaeger, Honeycomb, and
   DataDog all handle this differently; some silently drop spans, some produce malformed traces.
2. **Thread-safety violation.** `Entered` (the guard type) is `!Send`. If the outer async
   function is scheduled on the work-stealing pool, the compiler may accept the code (because
   the guard is never moved explicitly) while the runtime semantics are still incorrect.

**Scope.** The macro is defined and exported but appears unused in production code as of HEAD
(`grep -rn "#\[traced\]" crates/` finds only the declaration comment). However, it is part of
the published public API. Any downstream user who applies `#[traced]` to an async function will
get silent trace corruption in production.

**Correct approach.** For async functions, use `tracing::Instrument`:

```rust
use tracing::Instrument as _;

let expanded = if is_async {
    quote! {
        #fn_visibility #fn_sig {
            let span = tracing::debug_span!(#span_name);
            let start = ::std::time::Instant::now();

            // Instrument the inner future — span is entered/exited per-poll,
            // which is safe across .await points on any executor.
            let result = async move { #fn_body }
                .instrument(span)
                .await;

            let duration_ms = start.elapsed().as_millis();
            match &result {
                Ok(_)  => tracing::debug!(duration_ms, "span completed successfully"),
                Err(e) => tracing::warn!(duration_ms, error = %e, "span failed"),
            }
            result
        }
    }
};
```

For synchronous functions, `span.in_scope(|| { fn_body })` is correct (no guard escaping
across await since there are no await points):

```rust
} else {
    quote! {
        #fn_visibility #fn_sig {
            let span = tracing::debug_span!(#span_name);
            let start = ::std::time::Instant::now();
            let result = span.in_scope(|| { #fn_body });
            let duration_ms = start.elapsed().as_millis();
            match &result {
                Ok(_)  => tracing::debug!(duration_ms, "span completed successfully"),
                Err(e) => tracing::warn!(duration_ms, error = %e, "span failed"),
            }
            result
        }
    }
}
```

**Acceptance:**
- `grep -n "_guard = span.enter()" crates/fraiseql-observers-macros/src/lib.rs` → empty.
- Async path uses `.instrument(span)`.
- Sync path uses `span.in_scope(|| ...)`.

### Z2 — `fraiseql-observers-macros` has zero tests; the test block refers to non-existent examples

**File:** `crates/fraiseql-observers-macros/src/lib.rs`, lines 198–202

```rust
#[cfg(test)]
mod tests {
    // Macro tests are typically integration tests
    // See crate root for test examples
}
```

**Problems:**

1. There are no integration tests anywhere in the crate. `find crates/fraiseql-observers-macros
   -name "*.rs"` returns only `src/lib.rs`. The comment is a dead reference.
2. The two macros (`#[traced]`, `#[instrument]`) generate non-trivial token streams. The
   `#[traced]` macro has a correctness bug (Z1 above) that zero tests would have caught.
3. The `#[instrument]` macro silently ignores async bodies that do not return `Result` — the
   generated code compiles but the `match &result { Ok(_) => ... }` would fail for functions
   returning plain values. This is another latent bug.

**Fix:**

1. Delete the misleading comment.
2. Add a `tests/` directory with UI tests using `trybuild` or `macro_rules!` expansion tests:

```rust
// tests/traced_async.rs
use fraiseql_observers_macros::traced;

#[traced(name = "fetch_user")]
async fn fetch_user(id: u32) -> Result<String, String> {
    Ok(format!("user_{id}"))
}

#[tokio::test]
async fn traced_async_fn_completes() {
    let result = fetch_user(42).await;
    assert!(result.is_ok());
}
```

3. For the generated code correctness: restrict `#[traced]` and `#[instrument]` to functions
   that return `Result<_, _>`, or generate different output for non-Result return types.

**Acceptance:**
- `cargo test -p fraiseql-observers-macros` passes with at least one test for sync and one for
  async decorated functions.
- The dead comment is removed.

## Track AA — GitHub Actions Supply Chain (Priority: High)

The security-compliance workflow uses three third-party actions pinned to mutable branch
references (`@main`, `@master`). Any compromise of those upstream repositories would silently
inject malicious code into FraiseQL CI runs, with access to all CI secrets.

---

### AA1 — `trufflehog@main` and `trivy-action@master` are unpinned supply-chain risks

**File:** `.github/workflows/security-compliance.yml`, lines 36, 81, 99

```yaml
# Line 36 — secrets scan
uses: trufflesecurity/trufflehog@main

# Lines 81 and 99 — container security scan (run twice, for two Trivy formats)
uses: aquasecurity/trivy-action@master
```

**Why this matters.** Branch references resolve to whatever commit is at `HEAD` of `main`/`master`
at the time the workflow runs. A compromised or malicious push to either upstream repository
would execute arbitrary code in FraiseQL's CI environment with access to:
- `CARGO_TOKEN` (crates.io publish key)
- `GH_TOKEN` / `GITHUB_TOKEN` (repository write access)
- All other workflow secrets

This is a **supply-chain attack vector**. The CI job that scans for secrets is itself vulnerable
to a supply-chain attack that could exfiltrate the secrets it is protecting.

**Fix:** Pin each action to a full commit SHA. Fetch the current SHA from the action repository
and substitute it:

```yaml
# trufflehog — pin to a reviewed commit SHA
uses: trufflesecurity/trufflehog@<full-sha-here>  # v3.x.y

# trivy-action — pin to a reviewed commit SHA
uses: aquasecurity/trivy-action@<full-sha-here>  # v0.x.y
```

Add a comment documenting which tagged version the SHA corresponds to, so future maintainers
can update intentionally:

```yaml
uses: trufflesecurity/trufflehog@abc1234def5678...  # v3.88.1 — update SHA when bumping
```

**Also pin:** `dtolnay/rust-toolchain@master` (line 92 in `ci.yml`). `dtolnay/rust-toolchain`
is a widely trusted action and `@master` is its conventional reference (unlike `@stable` for
toolchains), but the principle still applies: pin to SHA for reproducibility.

**Acceptance:**
- `grep -rn "@main\|@master" .github/workflows/` → empty (or only in comments explaining the
  SHA it corresponds to).
- `grep -rn "trufflehog\|trivy-action" .github/workflows/` shows SHA-pinned refs.

### AA2 — `setup-python@v6` does not exist; workflow will fail at runtime

**Files:**
- `.github/workflows/generate-d2-diagrams.yml`, line 30
- `.github/workflows/release.yml`, line 297

```yaml
uses: actions/setup-python@v6
```

**Problem.** As of the most recent GitHub Actions release, `actions/setup-python` is at version
`v5`. `@v6` does not exist and will cause the workflow step to fail with an error like
`"Unable to resolve action actions/setup-python@v6"`. The D2 diagram generation workflow runs on

Both are currently silently broken.

**Fix:** Change both occurrences to the current latest major:

```yaml
uses: actions/setup-python@v5
```

Verify the correct current version in the [GitHub Actions Marketplace](https://github.com/marketplace/actions/setup-python)
before applying and pin to a SHA for supply-chain safety (consistent with AA1).

**Acceptance:**
- `grep -rn "setup-python@v6" .github/workflows/` → empty.
- Both workflows pass their setup-python step in a dry run.

## Track AB — SDK CI Coverage Gaps (Priority: Medium)

### AB1 — Seven official SDKs have no CI workflow; correctness is unverified

**The gap.** FraiseQL ships eleven official SDKs under `sdks/official/`. Only four have
dedicated CI workflows:

| SDK | CI workflow | Status |
|---|---|---|
| `fraiseql-csharp` | `csharp-sdk.yml` | ✅ Verified |
| `fraiseql-elixir` | `elixir-sdk.yml` | ✅ Verified |
| `fraiseql-fsharp` | `fsharp-sdk.yml` | ✅ Verified |
| `fraiseql-php` | `php-sdk.yml` | ✅ Verified |
| `fraiseql-dart` | ❌ None | Unverified |
| `fraiseql-go` | ❌ None | Unverified |
| `fraiseql-java` | ❌ None | Unverified |
| `fraiseql-python` | ❌ None | Unverified |
| `fraiseql-ruby` | ❌ None | Unverified |
| `fraiseql-rust` | ❌ None | Unverified |
| `fraiseql-typescript` | ❌ None | Unverified |

Seven SDKs are described as production-ready in their READMEs but are never built or tested
in CI. A schema-breaking change in `fraiseql-cli` or `schema.compiled.json` format would
silently break these SDKs with no automated detection.

**Priority by language usage:** Python and TypeScript are the primary authoring languages
(per `CLAUDE.md`). Go is explicitly documented. These three should be addressed first.

**Fix:** Create workflow files for each missing SDK following the pattern of the existing ones.
Minimum viable workflow for each:

```yaml
# .github/workflows/python-sdk.yml
name: Python SDK
on:
  push:
    paths: ['sdks/official/fraiseql-python/**', '.github/workflows/python-sdk.yml']
  pull_request:
    paths: ['sdks/official/fraiseql-python/**']
jobs:
  test:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: sdks/official/fraiseql-python
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
      - run: pip install uv && uv sync
      - run: uv run ruff check src
      - run: uv run pytest tests/
```

Each workflow must be path-filtered so it only runs when the SDK changes. Add a corresponding
publish step for the package registry (PyPI, npm, pkg.go.dev, Maven, crates.io) gated on
tag pushes following the pattern of `csharp-sdk.yml`.

**Acceptance:**
- `ls .github/workflows/*-sdk.yml | wc -l` → 11 (one per official SDK).
- Each SDK's test suite passes in CI without manual invocation.

### AB2 — Deprecated SDKs exist in both `sdks/community/` and `sdks/official/`, creating confusion

**Problem.** Three SDKs appear in both directories:

| SDK | In `sdks/official/` | In `sdks/community/` |
|---|---|---|
| `fraiseql-dart` | Full content, DEPRECATED.md | Full content, DEPRECATED.md |
| `fraiseql-elixir` | 5 files | 6 files (slight difference) |
| `fraiseql-ruby` | 1 file (`lib/`) only | Full content, DEPRECATED.md |

The `sdks/official/fraiseql-ruby` directory contains only a `lib/` subdirectory — it is an
empty shell. The community version has a complete gem structure. The `sdks/official/fraiseql-dart`
and `sdks/community/fraiseql-dart` are near-identical copies, both deprecated.

**Consequence:** The CI workflow for `elixir-sdk.yml` points to `sdks/official/fraiseql-elixir`
but there is also a `sdks/community/fraiseql-elixir` with slightly different content. A developer
editing the wrong copy will see CI pass while the "official" version is stale.

**Fix:**

1. For fully deprecated SDKs (`dart`, `ruby`): remove the `sdks/official/` copy entirely (or
   leave only a `REDIRECT.md` pointing to the community version or the HTTP API docs).
2. For `elixir`: reconcile the two directories. Since CI tests `sdks/official/fraiseql-elixir`,
   that is the canonical copy; remove or clearly mark `sdks/community/fraiseql-elixir` as
   archived.
3. Add a note to `sdks/README.md` explaining the `official/` vs `community/` distinction and
   the lifecycle for moving from one to the other.

**Acceptance:**
- `ls sdks/official/ sdks/community/ | sort | uniq -d` → empty (no directory name appears in
  both, or any duplicate has a clear redirect explaining the relationship).

## Track AC — Error Response Information Leakage (Priority: Medium)

### AC1 — `RuntimeError::IntoResponse` bypasses the server's `ErrorSanitizer`

**Files:**
- `crates/fraiseql-error/src/http.rs` (the `IntoResponse` impl)
- `crates/fraiseql-server/src/routes/graphql.rs` (where `ErrorSanitizer` is applied)

**Problem.** The server configures an `ErrorSanitizer` (loaded from `schema.compiled.json`
security config) that strips internal details from GraphQL error responses when enabled. However,
this sanitizer is applied only inside the GraphQL route handler. Auth and notification endpoints
use `RuntimeError`'s own `IntoResponse` impl, which calls `self.to_string()` directly as the
`error_description` field in the HTTP response body:

```rust
// fraiseql-error/src/http.rs:69
RuntimeError::Auth(e) => {
    // ...
    (status, ErrorResponse::new("authentication_error", self.to_string(), error_code))
    //                                                   ^^^^^^^^^^^^^^
    //                                           Includes full error message
}
```

The affected `self.to_string()` outputs include:

| Error variant | Response `error_description` |
|---|---|
| `AuthError::InvalidToken { reason }` | `"Invalid token: {reason}"` — may expose JWT parsing internals |
| `AuthError::ProviderError { provider, message }` | `"Provider error: {provider} - {message}"` — exposes provider name + internal message |
| `AuthError::AccountLocked { reason }` | `"Account locked: {reason}"` — exposes internal lock reason |
| `NotificationError::Configuration { message }` | `"Configuration error: {message}"` — exposes config details in 500 response |
| `NotificationError::Provider { provider, message }` | `"Provider error: {provider} - {message}"` — exposes internal infra names |

**Severity.** The `AccountLocked` and `ProviderError` variants are the most serious: they may
expose internal rate-limit reasons, provider URLs, or infrastructure identifiers to end users.
`NotificationError::Configuration` exposes config-level details in a 500 response.

**Note:** This is structurally distinct from the GraphQL-layer error sanitization already covered
in earlier plans. Those plans address GraphQL `errors` array sanitization. This finding concerns
the HTTP REST layer used by auth endpoints (`/auth/callback`, `/auth/refresh`, etc.) and
notification endpoints, which are not in the GraphQL path.

**Fix:**

Option A (minimal): In the `IntoResponse` impl, sanitize variants that include user-visible
details. Replace free-form `self.to_string()` with static descriptions for sensitive variants:

```rust
RuntimeError::Auth(e) => {
    let description = match e {
        // Sanitize variants that include internal details
        AuthError::InvalidToken { .. } => "The provided token is invalid.",
        AuthError::ProviderError { .. } => "Authentication provider error.",
        AuthError::AccountLocked { .. } => "This account is currently locked.",
        // Non-sensitive: expose the short message
        AuthError::TokenExpired => "Token has expired.",
        AuthError::SessionExpired => "Session has expired.",
        _ => "Authentication failed.",
    };
    (status, ErrorResponse::new("authentication_error", description, error_code))
}
```

Option B (preferred): Thread the `ErrorSanitizer` into the REST error path by wrapping the
`RuntimeError` at the route level, not in `fraiseql-error`. Routes that handle `RuntimeError`
should call `sanitizer.sanitize_rest_error(err)` before converting to HTTP response.

**Acceptance:**
- `curl -X POST /auth/refresh -H "..." -d '{"token": "expired"}'` → response body does not
  contain the raw provider name or internal lock reason.
- A test in `fraiseql-error/tests/http_responses.rs` asserts that `ProviderError` and
  `AccountLocked` responses do not expose the variant's internal fields.

## Track AD — Static Subscription Metrics Pollute Test Isolation (Priority: Low)

### AD1 — Module-level `AtomicU64` subscription counters reset only on process restart

**File:** `crates/fraiseql-server/src/routes/subscriptions.rs`, lines 57–60

```rust
static WS_CONNECTIONS_ACCEPTED:  AtomicU64 = AtomicU64::new(0);
static WS_CONNECTIONS_REJECTED:  AtomicU64 = AtomicU64::new(0);
static WS_SUBSCRIPTIONS_ACCEPTED: AtomicU64 = AtomicU64::new(0);
static WS_SUBSCRIPTIONS_REJECTED: AtomicU64 = AtomicU64::new(0);
```

**Problem.** These statics accumulate across all tests in the same process. A test that calls
`subscription_metrics()` after another test has already exercised the WebSocket handler will
see non-zero baseline values, making assertions like `assert_eq!(metrics.connections_accepted, 1)`
depend on test execution order. This was already observed in the `test_ws_metrics` family of tests
added in the quality plan.

Additionally, the exposed `SubscriptionMetrics` struct has no `active_connections` gauge field.
A Prometheus consumer watching `fraiseql_ws_connections_total` sees a monotonically increasing
counter but cannot determine how many connections are currently open. The field is named
`connections_accepted` (a reasonable counter name), but there is no corresponding decrement on
disconnect, making it impossible to derive "active connections" from the exported data.

**Fix (test isolation):**

```rust
#[cfg(test)]
pub fn reset_subscription_metrics_for_test() {
    WS_CONNECTIONS_ACCEPTED.store(0, Ordering::SeqCst);
    WS_CONNECTIONS_REJECTED.store(0, Ordering::SeqCst);
    WS_SUBSCRIPTIONS_ACCEPTED.store(0, Ordering::SeqCst);
    WS_SUBSCRIPTIONS_REJECTED.store(0, Ordering::SeqCst);
}
```

Call this at the start of each metrics-asserting test.

**Fix (active connections gauge):**

Add a `WS_CONNECTIONS_ACTIVE: AtomicI64 = AtomicI64::new(0)` that is incremented on accept
and decremented on disconnect:

```rust
// In handle_subscription (after accept):
WS_CONNECTIONS_ACTIVE.fetch_add(1, Ordering::Relaxed);

// In cleanup (after disconnect):
WS_CONNECTIONS_ACTIVE.fetch_sub(1, Ordering::Relaxed);
```

Expose it in `SubscriptionMetrics` and emit it as a Prometheus gauge:

```
# HELP fraiseql_ws_connections_active Currently open WebSocket connections
# TYPE fraiseql_ws_connections_active gauge
fraiseql_ws_connections_active 17
```

**Acceptance:**
- `reset_subscription_metrics_for_test()` exists and is called at the top of each
  subscription metrics test.
- `fraiseql_ws_connections_active` appears in `/metrics` output.
- `cargo test -p fraiseql-server -- test_ws_metrics` passes in any test execution order.
