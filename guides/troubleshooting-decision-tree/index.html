    Troubleshooting Decision Tree¶

    Quick diagnosis for common FraiseQL issues.

    üö® Problem Categories¶

    Choose your problem type:

    1. Installation & Setup

    3. GraphQL Queries

    5. Deployment

    1. Installation & Setup Issues¶

    ‚ùå "ModuleNotFoundError: No module named 'fraiseql'"¶

    Diagnosis: 

    pip show fraiseql

    If not installed: 

    pip install fraiseql

    If installed but still error: - ‚úÖ Check you're using correct Python environment - ‚úÖ Verify virtual environment activated: which python - ‚úÖ Reinstall: pip install --force-reinstall fraiseql

    ‚ùå "ImportError: cannot import name 'type' from 'fraiseql'"¶

    Diagnosis: - Check Python version: python --version - Required: Python 3.13+

    Fix: 

    # Upgrade Python
pyenv install 3.10
pyenv global 3.10

# Or use system package manager
sudo apt install python3.10  # Ubuntu
brew install python@3.10     # macOS

    ‚ùå "Rust pipeline not found" or "RustError"¶

    Diagnosis: 

    pip show fraiseql | grep Version

    Fix: 

    # Install with Rust support
pip install "fraiseql[rust]"

# Verify Rust pipeline
python -c "from fraiseql.rust import RustPipeline; print('Rust OK')"

    If still failing: - Rust compiler required for building - Install: https://rustup.rs/ - Then: pip install --no-binary fraiseql "fraiseql[rust]"

    2. Database Connection Issues¶

    Decision Tree¶

    ‚ùå Cannot connect to database
    |
    ‚îú‚îÄ‚Üí "Connection refused"
    |       ‚îî‚îÄ‚Üí PostgreSQL not running
    |           ‚îî‚îÄ‚Üí Start PostgreSQL: systemctl start postgresql
    |
    ‚îú‚îÄ‚Üí "password authentication failed"
    |       ‚îî‚îÄ‚Üí Check DATABASE_URL credentials
    |           ‚îî‚îÄ‚Üí Verify: psql ${DATABASE_URL}
    |
    ‚îú‚îÄ‚Üí "database does not exist"
    |       ‚îî‚îÄ‚Üí Create database: createdb fraiseql
    |
    ‚îî‚îÄ‚Üí "too many connections"
            ‚îî‚îÄ‚Üí Use PgBouncer connection pooler
                ‚îî‚îÄ‚Üí See: docs/production/deployment.md#pgbouncer

    ‚ùå "asyncpg.exceptions.InvalidPasswordError"¶

    Diagnosis: 

    # Test connection manually
psql postgresql://user:password@localhost/dbname

# If works, check environment variable
echo $DATABASE_URL

    Fix: 

    # Correct format:
export DATABASE_URL="postgresql://user:password@host:5432/database"

# Special characters in password? URL-encode them:
# @ ‚Üí %40, # ‚Üí %23, etc.

    ‚ùå "relation 'v_user' does not exist"¶

    Diagnosis: 

    -- Check if view exists
SELECT table_name FROM information_schema.tables
WHERE table_schema = 'public' AND table_name = 'v_user';

    Fix: 

    -- Create missing view
CREATE VIEW v_user AS
SELECT
    id,
    jsonb_build_object(
        'id', id,
        'name', name,
        'email', email
    ) as data
FROM tb_user;

    Prevention: - Run migrations: psql -f schema.sql - Check DDL Organization Guide

    3. GraphQL Query Issues¶

    Decision Tree¶

    ‚ùå GraphQL query fails
    |
    ‚îú‚îÄ‚Üí "Cannot query field 'X' on type 'Y'"
    |       ‚îî‚îÄ‚Üí Field not in GraphQL schema
    |           ‚îî‚îÄ‚Üí Check @type decorator includes field
    |
    ‚îú‚îÄ‚Üí "Variable '$X' of type 'Y' used in position expecting 'Z'"
    |       ‚îî‚îÄ‚Üí Type mismatch in query
    |           ‚îî‚îÄ‚Üí Fix variable type or make nullable: String | null
    |
    ‚îú‚îÄ‚Üí "Field 'X' of required type 'Y!' was not provided"
    |       ‚îî‚îÄ‚Üí Missing required field
    |           ‚îî‚îÄ‚Üí Add field or make optional in @input class
    |
    ‚îî‚îÄ‚Üí Query returns null unexpectedly
            ‚îî‚îÄ‚Üí Check PostgreSQL view returns data
                ‚îî‚îÄ‚Üí Run: SELECT data FROM v_table LIMIT 1;

    ‚ùå "Cannot return null for non-nullable field"¶

    Diagnosis: 

    import fraiseql

# Check type definition
@fraiseql.type(sql_source="v_user")
class User:
    id: int           # Required (non-nullable)
    name: str         # Required
    email: str | None # Optional (nullable)

    Fix:

    Option 1: Make field nullable in Python: 

    @fraiseql.type(sql_source="v_user")
class User:
    name: str | None  # Now nullable

    Option 2: Ensure PostgreSQL view never returns NULL: 

    CREATE VIEW v_user AS
SELECT
    id,
    jsonb_build_object(
        'id', id,
        'name', COALESCE(name, 'Unknown'),  -- Never null
        'email', email  -- Can be null
    ) as data
FROM tb_user;

    ‚ùå "Expected type 'Int', found 'String'"¶

    Diagnosis: - Type mismatch between GraphQL schema and PostgreSQL

    Fix:

    Python type ‚Üí PostgreSQL type mapping: - int ‚Üí INTEGER, BIGINT - str ‚Üí TEXT, VARCHAR - float ‚Üí DOUBLE PRECISION, NUMERIC - bool ‚Üí BOOLEAN - datetime ‚Üí TIMESTAMP, TIMESTAMPTZ

    Example fix: 

    import fraiseql

# Wrong
@fraiseql.type(sql_source="v_user")
class User:
    id: str  # PostgreSQL has INTEGER

# Correct
@fraiseql.type(sql_source="v_user")
class User:
    id: int  # Matches PostgreSQL INTEGER

    4. Performance Issues¶

    Decision Tree¶

    ‚ùå Queries are slow
    |
    ‚îú‚îÄ‚Üí N+1 query problem
    |       ‚îî‚îÄ‚Üí Use JSONB views with nested jsonb_agg
    |           ‚îî‚îÄ‚Üí See: performance/index.md#n-plus-one
    |
    ‚îú‚îÄ‚Üí Missing database indexes
    |       ‚îî‚îÄ‚Üí Add indexes on foreign keys and WHERE clauses
    |           ‚îî‚îÄ‚Üí CREATE INDEX idx_post_user_id ON tb_post(user_id);
    |
    ‚îú‚îÄ‚Üí Large result sets
    |       ‚îî‚îÄ‚Üí Implement pagination
    |           ‚îî‚îÄ‚Üí Use LIMIT/OFFSET or cursor-based
    |
    ‚îî‚îÄ‚Üí Connection pool exhausted
            ‚îî‚îÄ‚Üí Use PgBouncer
                ‚îî‚îÄ‚Üí See: production/deployment.md#pgbouncer

    ‚ùå "Too many connections to database"¶

    Diagnosis: 

    -- Check current connections
SELECT count(*) FROM pg_stat_activity;
SELECT max_connections FROM pg_settings WHERE name = 'max_connections';

    Immediate fix: 

    -- Kill idle connections
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'idle' AND state_change < now() - interval '5 minutes';

    Permanent fix:

    Install PgBouncer: 

    # Docker Compose
services:
  pgbouncer:
    image: pgbouncer/pgbouncer
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/fraiseql
      - POOL_MODE=transaction
      - DEFAULT_POOL_SIZE=20
    ports:
      - "6432:6432"

# Update DATABASE_URL to use PgBouncer
DATABASE_URL=postgresql://user:pass@pgbouncer:6432/fraiseql

    5. Deployment Issues¶

    ‚ùå "Health check failing in Kubernetes"¶

    Diagnosis: 

    # Check pod logs
kubectl logs -f deployment/fraiseql-app -n fraiseql

# Test health endpoint manually
kubectl port-forward deployment/fraiseql-app 8000:8000 -n fraiseql
curl http://localhost:8000/health

    Common causes:

    1. Database not ready: 

      # Add initContainer to wait for database
initContainers:
- name: wait-for-db
  image: busybox
  command: ['sh', '-c', 'until nc -z postgres 5432; do sleep 1; done']

    2. Wrong DATABASE_URL: 

      # Check secret
kubectl get secret fraiseql-secrets -n fraiseql -o yaml
echo "BASE64_STRING" | base64 -d

    3. Not enough resources: 

      resources:
  requests:
    memory: "256Mi"  # Increase if OOMKilled
    cpu: "250m"

    ‚ùå "Container keeps restarting"¶

    Diagnosis: 

    # Check exit code
kubectl describe pod <pod-name> -n fraiseql

# Common exit codes:
# 137 ‚Üí OOMKilled (increase memory)
# 1   ‚Üí Application error (check logs)
# 143 ‚Üí SIGTERM (graceful shutdown, normal)

    Fix: 

    # Increase memory limit
resources:
  limits:
    memory: "1Gi"  # Was 512Mi

# Add startup probe (more time to start)
startupProbe:
  httpGet:
    path: /health
    port: 8000
  failureThreshold: 30  # 30 * 5s = 150s max startup
  periodSeconds: 5

    6. Authentication Issues¶

    ‚ùå "@authorized decorator not working"¶

    Diagnosis: 

    # Check if user context is set
import fraiseql

@authorized(roles=["admin"])
@fraiseql.mutation
class DeletePost:
    async def resolve(self, info):
        # Check context
        print(f"User: {info.context.get('user')}")
        print(f"Roles: {info.context.get('roles')}")

    Fix:

    Ensure context middleware sets user: 

    from fraiseql.fastapi import create_fraiseql_app

async def get_context(request):
    # Extract JWT token
    token = request.headers.get("Authorization", "").replace("Bearer ", "")

    # Decode token
    user = decode_jwt(token)

    # Return context with user and roles
    return {
        "user": user,
        "roles": user.get("roles", []),
        "request": request
    }

app = create_fraiseql_app(
    ...,
    context_getter=get_context
)

    ‚ùå "Row-Level Security blocking queries"¶

    Diagnosis: 

    -- Check RLS policies
SELECT tablename, policyname, cmd, qual
FROM pg_policies
WHERE schemaname = 'public';

-- Test as specific user
SET ROLE tenant_user;
SELECT * FROM tb_post;  -- Should only see tenant's posts

    Fix:

    If no rows returned when expected: 

    -- Check if policy is correct
ALTER POLICY tenant_isolation ON tb_post
USING (tenant_id = current_setting('app.current_tenant_id')::UUID);

-- Ensure tenant_id is set
SET app.current_tenant_id = 'tenant-uuid-here';

-- Test again
SELECT * FROM tb_post;

    Before Opening an Issue¶

    1. Search existing issues: GitHub Issues
    2. Check discussions: GitHub Discussions
    3. Review documentation: Complete Docs

    Opening a Good Issue¶

    Include: - FraiseQL version: pip show fraiseql | grep Version - Python version: python --version - PostgreSQL version: psql --version - Minimal reproduction: smallest code that reproduces issue - Error messages: Full stack trace - What you've tried: Show troubleshooting steps attempted

    Template: 

    ## Environment
- FraiseQL: 1.0.0
- Python: 3.10.5
- PostgreSQL: 16.1
- OS: Ubuntu 22.04

## Issue
[Clear description of problem]

## Reproduction
\```python
# Minimal code to reproduce
\```

## Error
\```
Full error message
\```

## Attempted Fixes
- Tried X, result: Y
- Tried Z, result: W
