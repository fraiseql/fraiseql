# FraiseQL Full Repository Audit — 2026-06-11

## Executive Summary

This audit covers security, code quality, and dependencies across the FraiseQL v2 Rust workspace, its SDKs, and deployment artifacts. It surfaced one **critical** SQL-injection vulnerability in the MySQL mutation path, a cluster of **high-severity** authorization bypasses (Relay `node(id:)`, federation `_entities`, observer/DLQ admin API, OIDC anonymous-mode admin routes), several **high-severity** silently-inert security controls (token revocation never read, error sanitization unmounted on REST, field-encryption write half never wired, audit logging in-memory only), multiple **high-severity** remotely-triggerable panics and DoS vectors in the REST `?select=` parser and the GraphQL complexity validator, and a broad pattern of "documented-but-dead" code (saga orchestration that fabricates success, observer action stubs reporting success, SDK `config()` no-ops, and an SDK publishing pipeline frozen at 2.1.6 since v2.2.1). Dependency posture is mostly clean but carries a feature-gated EOL TLS stack with three risk-accepted advisories whose self-imposed deadline lapses 2026-06-15, a broken `metrics` facade, and several dead/drifted manifest entries.

## Methodology

34 parallel finder agents (17 security lenses, 14 quality reviewers, 3 dependency auditors) read the codebase statically. Every critical/high security finding was re-verified by a 3-lens adversarial panel (kept only if ≥2 lenses upheld it under code-reality, exploitability, and mitigation lenses); medium security and critical/high quality findings were verified by a single adversarial skeptic; low-severity items are unverified hygiene notes labeled `unverified-low`. **5 candidate findings were refuted and dropped** during verification. Where multiple lenses reported the same underlying defect, the findings are merged below with all contributing finders noted. The audit was entirely static — no server boot, fuzzing, or runtime observation was performed (see Coverage Gaps).

Severity counts after merges: **1 critical, ~40 high, ~30 medium, ~55 low/unverified.**

## CRITICAL

### C1. SQL injection in MySQL stored-procedure mutation path (backslash escape breakout)

**File:** `crates/fraiseql-db/src/mysql/adapter.rs:195-197` · **Finders:** db-layer, db (merged) · **Verification:** panel-3-lens + single-skeptic

`execute_function_call` builds `CALL fn(...)` via `sqlx::raw_sql` (text protocol, multi-statement capable) with arguments inline-escaped by `mysql_escape_json_value`, which only doubles single quotes (`s.replace('\'', "''")`) and does **not** escape backslashes. The MySQL adapter never sets `sql_mode`, so connections run in default mode where backslash is a string-literal escape character. An argument like `\', <payload>; -- ` escapes to `'\'', <payload>; -- '`; MySQL parses `\'` as an escaped quote, the next `'` closes the literal, and the remainder executes as raw SQL. sqlx-mysql 0.8.6 negotiates `MULTI_STATEMENTS`, so `DROP TABLE` payloads are viable. Arguments flow directly from GraphQL mutation input (`runners/mutation/mod.rs` → `execute_function_call_with_session`, whose non-PG default delegates here). MySQL is a documented supported backend wired in `main.rs:663`. The function's doc comment is inverted — it claims quote-doubling is "safe regardless of NO_BACKSLASH_ESCAPES," which is backwards. The aggregation path in the same codebase escapes backslashes correctly, proving this is an omission.

**Evidence:** `serde_json::Value::String(s) => format!("'{}'", s.replace('\'', "''"))` then `sqlx::raw_sql(&call_sql).fetch_all(&self.pool)`.

**Recommendation:** Bind arguments as parameters (`?` placeholders / bound CALL) instead of inline escaping. If text protocol is required, escape backslashes (`\\` and `'`→`''`) AND force `SET sql_mode = ...,NO_BACKSLASH_ESCAPES`; reject `;` in the assembled statement.

## HIGH

### H1. SQL injection via unquoted aggregation GROUP BY alias from GraphQL variables

**File:** `crates/fraiseql-core/src/runtime/aggregation/expressions.rs:33` (dup: `partial_period_builder.rs:177,183`) · **Finder:** core-sql-injection · **Verification:** panel-3-lens

The GROUP BY SELECT alias is interpolated raw: `format!("{} AS {}", column, alias)` with no `quote_identifier` and no validation, unlike the JSONB path (single-quote escaped) and ORDER BY (quoted). The alias is the raw GraphQL variables JSON object key (`aggregate_parser.rs:284-287` `GroupBySelection::Dimension { path: key.clone(), alias: key.clone() }`), and JSON keys are unconstrained by GraphQL Name grammar. The allowlist guard (`compiler/aggregation.rs:404-405`) only fires when `dimensions.paths` is non-empty; `detector.rs:121-131` falls back to `Vec::new()` when the source table is empty at compile time (a realistic CI/dev default) while still registering the dimensions column. A groupBy key like `a, (SELECT string_agg(rolpassword,',') FROM pg_authid) AS leak` appends an uncorrelated scalar-subquery column to the SELECT list, enabling cross-tenant/cross-table exfiltration. Runs anonymously; RLS only affects WHERE/session vars, not the SELECT alias.

**Recommendation:** Route the alias through `quote_identifier` or validate against `[_A-Za-z][_0-9A-Za-z]*` at parse time; do not skip the allowlist when `dimensions.paths` is empty.

### H2. Relay `node(id:)` global lookup bypasses RLS, inject_params, and requires_role

**File:** `crates/fraiseql-core/src/runtime/executor/runners/query_relay.rs:410,445,466` · **Finders:** core-authz, core (merged) · **Verification:** panel-3-lens + single-skeptic

`execute_node_query` receives no `SecurityContext` (both `security.rs:280` authenticated and `execution.rs:155` anonymous dispatch drop it) and builds a WHERE clause of only `data->>'id' = uuid`. It never evaluates `RLSPolicy`, never applies `inject_params` (tenant/owner filters), and never enforces `requires_role`. The only guard, `type_has_gated_field` (line 445), covers field-level `authorize` markers only. `node_type_index` is built from every query with a `sql_source` (`core.rs:137-142`), so any role-restricted/inject-filtered type is reachable via `node(id: base64("Type:uuid"))`. The regular query path enforces all three controls — confirming a genuine bypass. Classic IDOR: a JWT-holding or anonymous client reads another tenant's/admin-only row given a leaked UUID. In-code comment (lines 490-493) admits the gap as follow-up #329.

**Recommendation:** Thread `Option<&SecurityContext>` into `execute_node_query`; enforce `requires_role`, AND the RLS WHERE onto the id filter, and apply `inject_params`. Until then fail closed when RLS/requires_role/inject_params is configured or no context is present.

### H3. Federation `_entities` resolution escapes key values with single-quote-only escaping (unsafe on MySQL)

**File:** `crates/fraiseql-federation/src/sql_utils.rs:100` · **Finder:** federation-functions · **Verification:** panel-3-lens

Client `representations` key-field values flow `value_to_string` → `escape_sql_string` (`value.replace('\'', "''")`) → `format!("'{}'", ...)` → `adapter.execute_raw_query(&sql)` with no bind parameters (`database_resolver.rs:138`). Correct for PostgreSQL/SQL Server (standard_conforming_strings) but unsafe on MySQL (default backslash-escape mode): a lone backslash consumes the closing quote, letting a second value inject SQL. `traits.rs:579-581` documents that other paths migrated to bind params specifically to retire `escape_sql_string`; federation never did. One verifier noted clean exploitation on MySQL additionally requires `ANSI_QUOTES` (the double-quoted table name is otherwise a syntax error), narrowing realistic exploitation toward medium, but the core technical defect (unparameterized, backend-incorrect escaping of attacker input on a supported backend) holds.

**Recommendation:** Use parameterized execution (`$N`/`?`/`@P1`) as the aggregate path does; if a literal path must remain, make the escaper backend-aware.

### H4. Exponential fragment-spread amplification in the depth/complexity validator (DoS)

**File:** `crates/fraiseql-core/src/graphql/complexity.rs:387` (also alias counter, line 559 — low) · **Finder:** core-input-dos · **Verification:** panel-3-lens

`selection_set_depth`/`selection_set_complexity` re-walk each fragment spread with no memoization and no per-fragment visited set. `recursion_depth` increments only on a spread hop, so the `> 32` guard bounds chain depth but not branching factor → B^D recursive calls. A ~1 KB query with 31 chained fragments (depth 30, branching 2, all under the guard) forces ~2.1 billion calls (~88 s single-threaded). The full metric is computed before any limit comparison, so the validation step itself is the DoS — configured limits never reject it. Runs synchronously on the tokio worker (`handler.rs:397`), enabled by default; the 1 MB body limit leaves room for far worse tunings. `TimeoutLayer` is opt-in and cannot preempt synchronous CPU-bound code.

**Recommendation:** Memoize each fragment's depth/complexity by name (resolve in topological order, treat cycles as over-limit), or carry a global node-visit budget and abort when exceeded.

### H5. OIDC `required = false` silently disables auth on observer/design/introspection admin endpoints

**File:** `crates/fraiseql-server/src/middleware/oidc_auth.rs:140` · **Finder:** server-http-authnz · **Verification:** panel-3-lens

`oidc_auth_middleware` lets tokenless requests through whenever the shared validator reports `is_required() == false` (a documented anonymous-access mode for public RLS-backed APIs). The single validator is shared between the data plane and admin routers (observers, design-audit, introspection, schema export, metadata, playground). Admin routers only check that a validator *exists* (`observers.rs:50`), never `is_required()`. Result: with `required=false`, `GET /api/observers/{id}` returns the full observer config including Webhook `actions[].headers` secrets, and POST/PATCH/DELETE allow anonymous observer creation (SSRF/exfiltration). Per-endpoint `*_require_auth = true` flags are silently defeated.

**Recommendation:** Gate admin-router mounting on `validator.is_required()` (refuse + WARN when false), or wrap admin routes in a variant that always requires a valid token; enforce `*_require_auth` independently of the global `required` flag.

### H6. Observer & DLQ admin API is authenticated but not authorized — any valid OIDC token grants cluster admin and reads webhook secrets

**File:** `crates/fraiseql-server/src/server/routing/observers.rs:74` · **Finder:** observers · **Verification:** panel-3-lens

All observer admin routers (management, changelog, runtime, DLQ) are gated only by `oidc_auth_middleware`, which does authentication but no scope/role check. No observer/DLQ handler extracts `AuthUser` or checks scopes. The same validator authenticates end users on `/graphql`, so in the common single-issuer SaaS deployment every authenticated end user can read all observers (incl. literal bearer-token secrets in `actions[].headers`), create/update/delete observers, reload the runtime, and drive DLQ retry-all/delete. The codebase has the proper primitive (`require_admin_scope`/`fraiseql:admin`, used by the RBAC admin router) but deliberately does not apply it here. The #348 fix closed anonymous writes, not authenticated-user escalation.

**Recommendation:** Require a dedicated admin scope/role (e.g. `observers:admin`) after authentication, or mount the admin API behind a separate operator-only validator/issuer.

### H7. Error-sanitization middleware never mounted — REST leaks raw DB error text even when sanitization is enabled

**File:** `crates/fraiseql-server/src/middleware/error_sanitization.rs:37` · **Finders:** server-leak-headers, server (merged) · **Verification:** panel-3-lens + single-skeptic

The module is not even declared in `middleware/mod.rs` (orphaned, never compiled) and `apply_middleware` never layers it. GraphQL sanitizes inline at call sites, but the REST path has zero sanitization: `From<FraiseQLError> for RestError` maps non-22/23-SQLSTATE Database errors to `RestError::internal(err.to_string())` (`response.rs:149,152`), and `error_response` writes `e.message` verbatim into the body — leaking schema names, constraint details, SQL fragments from any 42xxx/P0001/XX000/connection error. Even if mounted, `sanitize_json_error` only matches a top-level `errors` array or `message`/`code`, while REST bodies nest under `{"error":{...}}`, so it would no-op. Affects builds with the `rest` feature + sanitization enabled.

**Recommendation:** Declare and mount the middleware (fixing its REST body-shape handling) or add explicit `ErrorSanitizer` calls to the REST error-rendering site; add a REST integration test forcing a non-22/23 DB error.

### H8. Token revocation is write-only — revoked access tokens are never rejected

**File:** `crates/fraiseql-server/src/middleware/oidc_auth.rs:158` · **Finder:** auth-jwt-pkce · **Verification:** panel-3-lens

`[security.token_revocation]` config, three backends, and `POST /auth/revoke[-all]` write revoked `jti`s, but the read side (`TokenRevocationManager::check_token`/`RevocationStore::is_revoked`) has zero non-test callers. The auth middleware validates the JWT and decodes `jti` into `SessionJti` but never consults the revocation store. The doc comment's "(when enabled) replay-cache check" refers to fraiseql-core's separate single-use jti replay cache, not the operator-facing revocation store. Revoking a token (logout, compromise, admin action) is a silent no-op; tokens work until natural `exp`.

**Recommendation:** Thread `TokenRevocationManager` into `OidcAuthState` and call `check_token(jti)` after validation; return 401 on `Revoked`/`MissingJti`/`StoreUnavailable` (when `fail_open=false`). Add a request-path revocation test.

### H9. Storage uploads authorized at bucket level, not object level — any authenticated user can overwrite or pre-claim another user's object

**File:** `crates/fraiseql-storage/src/rls/mod.rs:59`; `routes/mod.rs:192` · **Finders:** storage, storage (merged) · **Verification:** panel-3-lens + single-skeptic

`can_write` only checks authenticated-or-admin (ignores object/owner); `put_handler` calls only `can_write` before uploading; the metadata upsert `ON CONFLICT DO UPDATE` deliberately does **not** update `owner_id`. In a shared Private bucket with predictable keys: (1) integrity — user B overwrites A's key, content becomes B's bytes while metadata still attributes it to A; (2) confidentiality — B pre-creates a key (owner=B), A's later upload keeps `owner_id=B`, locking A out of their own data while B can read it. Delete is correctly owner-scoped (`can_delete`), proving the write-side gap is an oversight. No cross-user overwrite test exists.

**Recommendation:** On upsert conflict, reject when existing `owner_id` differs (and not admin), or treat overwrite as a distinct per-object permission; never silently preserve the original owner. Add a non-owner-overwrite test.

### H10. VaultBackend::rotate_secret self-deadlocks on its own per-secret lock

**File:** `crates/fraiseql-secrets/src/secrets_manager/backends/vault/backend.rs:266` · **Finders:** secrets, secrets (merged) · **Verification:** panel-3-lens + single-skeptic

`rotate_secret` holds the per-secret non-reentrant tokio `Mutex` via `with_secret_lock`, invalidates the cache, then calls `get_secret_with_expiry`, which misses the just-invalidated cache and re-acquires the same `Arc<Mutex<()>>` → permanent self-deadlock on first invocation. `LeaseRenewalTask::renew_expiring_leases` drives renewal through this path, so the first near-expiry lease wedges the renewal loop forever and dynamic credentials silently expire. No test calls `rotate_secret`. Reachable by external consumers of the published crate (the server binary itself only constructs the Env backend, limiting in-product blast radius).

**Recommendation:** Extract the lock-free fetch+cache body into a private helper called by both methods while holding the lock once. Add a timeout-wrapped regression test.

### H11. Webhook `body_template` does unescaped string substitution then re-parses as JSON — JSON/payload injection into signed webhooks

**File:** `crates/fraiseql-observers/src/actions.rs:440` · **Finder:** observers · **Verification:** panel-3-lens

`render_body_template` substitutes event field values via raw `s.clone()` (no JSON escaping) then `serde_json::from_str`. Since the feature embeds entity-row string fields (usernames, comments) into a JSON document, an attacker controlling a string field can break out of the surrounding string and inject/override keys (e.g. `"verified": true`). The injected body is then HMAC-signed and delivered with `X-FraiseQL-Signature-256`, so a downstream verifier trusts the forged fields. Slack/email paths are not affected (serde `json!` escaping / plain text). The README even recommends the vulnerable pattern.

**Recommendation:** Do not build JSON by string interpolation — substitute leaf placeholders as serde-escaped JSON values, or build the payload as a `serde_json::Value` programmatically; remove the re-parse-arbitrary-text step.

### H12. Field-encryption write half never wired — mutations write plaintext, reads then fail to decrypt

**File:** `crates/fraiseql-secrets/src/encryption/middleware.rs:207` · **Finder:** secrets · **Verification:** single-skeptic

`FieldEncryptionService::encrypt_variables` has zero callers outside the crate's tests. The server wires the service but only calls `decrypt_response` on the read path; the core mutation executor never consults `field.encryption`. Any field marked `encryption` is written to the DB in plaintext, and the subsequent read attempts base64-decode + AES-GCM decrypt of plaintext, returning "Field decryption failed" (500). DB-side encryption is impossible (keys live server-side). The advertised at-rest encryption cannot work end-to-end as shipped.

**Recommendation:** Wire `encrypt_variables` into the mutation path before SQL parameter binding, or remove the dead encrypt half and document that encryption-on-write is unsupported.

### H13. Audit logging is dead code — `AuditLogger` is in-memory only, never wired to any secret/encryption operation

**File:** `crates/fraiseql-secrets/src/encryption/audit_logging.rs:239` · **Finder:** secrets · **Verification:** single-skeptic

Docs and the project advertise "Audit Logging — track all secret access for compliance (HIPAA/PCI-DSS/GDPR/SOC 2)," but `AuditLogger` stores entries in a `Vec` commented "for testing," has no persistence/tracing sink, and is never invoked from `VaultBackend`, `SecretsManager`, `DatabaseFieldAdapter`, `FieldEncryptionService`, or the server. (Note: fraiseql-auth has a *separate*, genuinely-wired audit system covering auth-side secrets; it does not cover fraiseql-secrets' Vault/encryption operations, which is what the compliance claim references.)

**Recommendation:** Wire `AuditLogEntry` emission into `get_secret`/`rotate_secret` and the encrypted-field adapter, or remove the compliance claims until implemented.

### H14. Vault Transit encrypt/decrypt uses unpadded base64; real Vault requires padded standard base64

**File:** `crates/fraiseql-secrets/.../vault/backend.rs:753,809` · **Finder:** secrets · **Verification:** single-skeptic

`encrypt_field` encodes plaintext with `STANDARD_NO_PAD` (Vault's Go `base64.StdEncoding` rejects unpadded for lengths not divisible by 3); `decrypt_field` decodes Vault's always-padded response with `STANDARD_NO_PAD` (errors on trailing `=`). Both Transit ops fail against a real Vault for ~2/3 of input lengths. The only Transit test checks a size guard, never a round-trip. (No in-workspace caller; the product path uses local AES-GCM. Affects published-crate consumers; fails loud, no data corruption.)

**Recommendation:** Use padded `STANDARD` for both directions; add a wiremock round-trip test.

### H15. AppRole login posts role_id/secret_id before SSRF address validation

**File:** `crates/fraiseql-secrets/.../vault/backend.rs:337-346` · **Finders:** secrets, secrets (merged) · **Verification:** single-skeptic (medium-rated, listed here under secrets cluster)

`with_approle` POSTs the AppRole credentials to `{addr}/v1/auth/approle/login` and only afterwards calls `Self::new(addr, ...)`, which runs `validate_vault_addr`. With an attacker-influenced/misconfigured address (e.g. 169.254.169.254) the high-value `secret_id` is delivered before the guard fires. The token path validates first; only AppRole has this ordering bug. *(Severity: medium.)*

**Recommendation:** Call `validate_vault_addr(addr)?` as the first statement of `with_approle`.

### H16. Server constructors drifted — relay & Arrow paths drop schema-format validation, #421 max_page_size, and mutation audit logging

**File:** `crates/fraiseql-server/src/server/extensions.rs:128,236` · **Finder:** server · **Verification:** single-skeptic

`Server::new` is the only constructor that validates `schema.validate_format_version()`, reads compiled `audit_logging_enabled` into `audit_mutations`, and applies the #421 page-size precedence. `with_relay_pagination` (`Executor::new_with_relay`) and `with_flight_service` (`Executor::new`) both use `RuntimeConfig::default()` — booting without format-version checking, with `audit_mutations=false` (compiled audit logging silently no-ops), and `max_page_size` pinned to default (operator/env/compiled overrides ignored). These are mainline: `main.rs:621` selects relay for any schema with a relay query; `main.rs:594` selects flight for every arrow build. `with_flight_service` also hand-duplicates ~60 lines of struct init.

**Recommendation:** Extract the schema→RuntimeConfig derivation into one shared helper used by all three constructors; use `Executor::with_config_and_relay` on the relay path. Add a regression test.

### H17. Remotely-triggerable panic in REST `?select=` parser — char-index used to byte-slice a `&str`

**File:** `crates/fraiseql-server/src/routes/rest/params/select.rs:63` (also 82, 117) · **Finder:** panic-surface · **Verification:** panel-3-lens (reproduced)

`parse_select_entries_inner` collects `Vec<char>`, walks it with a `usize` index `i`, then byte-slices the original `&str` with char-position indices. Any multi-byte UTF-8 char before the cut point desynchronizes the indices and panics ("byte index N is not a char boundary"). The value comes from the percent-decoded `?select=` query param, parsed before `validate_field_name` runs. `/<resource>?select=%C3%A9` (decodes to 'é') reaches the slice and panics. No CatchPanic middleware; per-request task aborts (per-request DoS / log noise). REST is a non-default feature.

**Recommendation:** Slice via char-aware boundaries (`chars[a..b].iter().collect::<String>()`, byte-offset tracking, or `char_indices()`) at all three sites.

### H18. MAX_PARSE_DEPTH recursion guard defeated by variable shadowing → unbounded recursion / stack overflow on `?select=`

**File:** `crates/fraiseql-server/src/routes/rest/params/select.rs:121` · **Finder:** panic-surface · **Verification:** panel-3-lens

Inside the embedded-resource branch, `let mut depth = 1;` (line 101) shadows the recursion-depth parameter; the paren-matching loop drives the shadow to 0, so the recursive call `parse_select_entries_inner(inner, depth + 1)` always passes `1`. The `> MAX_PARSE_DEPTH` guard never fires regardless of nesting. `validate_embedding_depth` runs only after parsing completes. A `?select=a(a(a(...)))` ~30 KB query (well under the 1 MiB query-string cap, far over the ~2 MiB worker stack) overflows the stack — a Rust stack overflow aborts the whole process (SIGSEGV), full-server DoS. REST is a non-default feature.

**Recommendation:** Rename the paren counter (e.g. `paren_depth`) and pass the real recursion depth; add a >32-level regression test.

### H19. Authenticated multi-root queries silently drop every root field except the first

**File:** `crates/fraiseql-core/src/runtime/executor/security.rs:236` · **Finder:** core · **Verification:** single-skeptic

The anonymous dispatcher fans out multi-root queries via `execute_parallel`; the authenticated dispatcher (`execute_with_security_internal`, the path fraiseql-server routes all authenticated requests through) has no `is_multi_root` branch — `QueryType::Regular` matches only the first selection. `{ users { id } posts { id } }` returns `{"data":{"users":[...]}}` and silently omits `posts`, no error. The #422 authorizer layer on this same function explicitly collects one op per root, proving multi-root is in-scope — yet execution serves only the first. A `fraiseql_multi_root_queries_total` metric exists that only the anonymous path can increment.

**Recommendation:** Add the same multi-root fan-out (threading the SecurityContext) to `execute_with_security`, or refactor both entry points onto one shared dispatcher.

### H20. Remotely-triggerable panics: fixed byte-offset string slicing on user-controlled UTF-8 (6 sites)

**File:** `crates/fraiseql-core/src/runtime/executor/execution.rs:52`; `security.rs:193`; `query_tracing.rs:99,277`; `audit_export_syslog.rs:121`; `error_formatter.rs:326` · **Finder:** core · **Verification:** single-skeptic

Error/log paths truncate user-controlled strings with `&s[..N]`, panicking on non-char-boundary bytes. The query-timeout snippet (`&query[..100]`) is copy-pasted in `execution.rs` and `security.rs` (live request path); an attacker sends a slow query with a multi-byte char at byte 99-100 so the timeout handler itself panics. The audit-export site lets an attacker suppress their own audit record. Tasks are tokio-isolated (per-request abort, not process crash); 2 of the 6 sites (`query_tracing`) are test-only.

**Recommendation:** Add one char-boundary-safe truncation helper and route all six sites through it; delete the duplicated timeout snippet.

### H21. `lint --fail-on-critical` / `--fail-on-warning` never affect the exit code — CI gates silently pass

**File:** `crates/fraiseql-cli/src/commands/lint.rs:189` · **Finder:** cli · **Verification:** single-skeptic

When the fail flag trips, `lint::run` returns `Ok(CommandResult::error(...))`; the runner's Lint arm maps every `Ok` to `Ok(())` without inspecting `result.status`, so the process exits 0. The clap help promises a non-zero exit. Tests only cover clean-schema exit-0 and file-not-found. Any CI using `fraiseql lint --fail-on-critical` as a gate passes regardless of findings. `Commands::Validate` does check `status` — Lint is the odd one out.

**Recommendation:** In `runner.rs`, check `result.status` and `process::exit(2)`; add a dirty-schema exit-code test.

### H22. `federation check` exits 0 when composition validation fails

**File:** `crates/fraiseql-cli/src/commands/federation/check.rs:189` · **Finder:** cli · **Verification:** single-skeptic

`check::run` builds `status: "validation-failed"`/`COMPOSITION_ERROR` but returns `Ok(result)`; the runner maps it to `Ok(())`, so the deploy-gating command always exits 0 unless the schema file is unreadable. `Commands::Validate` does the missing check — federation check omits it. No test asserts a non-zero exit.

**Recommendation:** Mirror the Validate pattern — return an error / exit 2 when `status == "validation-failed"`; add a failing-composition exit-code test.

### H23. `fraiseql serve fraiseql.toml` overwrites the user's source TOML with compiled JSON

**File:** `crates/fraiseql-cli/src/commands/serve.rs:110` · **Finder:** cli · **Verification:** single-skeptic (reproduced)

`compile_schema` derives output via `input.replace(".json", ".compiled.json")` — a no-op for a `.toml` input, so `output == input` and `compile::run` ends at `fs::write(output, output_json)`, destroying the source `fraiseql.toml` (the primary authoring workflow). Reproduced live: running `serve fraiseql.toml` replaced the file with compiled JSON. The command is `#[command(hide = true)]` but fully dispatchable via `runner.rs`. `String::replace` also corrupts paths with a `x.json/` component; the `port` arg is decorative.

**Recommendation:** Reject non-`.json` inputs (or bail if `output == input`) before writing; better, delete `serve` — `run --watch` already provides hot-reload with a real server.

### H24. SMS/Push/Search/Cache observer actions are stubs that report success while doing nothing

**File:** `crates/fraiseql-observers/src/actions_additional.rs:50` (+103, 171, 233) · **Finder:** observers · **Verification:** single-skeptic

`ActionConfig` is tagged `type`, so `type = "sms"/"push"/"search"/"cache"` deserializes and dispatches, but every `execute()` fabricates `success: true` with a fake message id and sends nothing. `config.validate()` accepts these variants; the runtime always uses `DefaultActionDispatcher`. ## COVERAGE GAPS

The completeness critic identified the following audit blind spots (prioritized):

1. **SQL semantics (no lens).** 128 `.sql` files (`migrations/` trinity/clickhouse/elasticsearch, `sql/helpers/`, `tests/sql`, `examples/*/sql`) were not audited. No lens checked GRANTs, SECURITY DEFINER, trigger correctness, or whether tenant_id/RLS stamping in the trinity/change-spine migrations matches runtime assumptions — tenant-leak and data-integrity risks live in SQL, not crates.

2. **Compiler→runtime contract drift (structurally invisible to per-crate lenses).** `fraiseql-cli`/`fraiseql-codegen` emit `schema.compiled.json`; server/core parse it with serde defaults (e.g. `InputFieldDefinition.nullable` default-true; the `token_revocation: null` boot-refusal incident). A field added on one side and silently defaulted on the other passes both crates' lenses — this class has caused dev-RED twice. No lens diffed the emit schema against the parse schema.

3. **Deployment/ops hardening semantics.** Beyond secret-grepping, nobody audited container securityContext, exposed ports, nginx TLS config, crowdsec scenarios, or whether prod compose defaults are safe when `${DB_PASSWORD}` is unset, across `deploy/`, `k8s/`, six `docker/docker-compose.*.yml`, the root `Dockerfile`, `grafana/`, `monitoring/`.

4. **Examples as shipped insecure patterns.** 26 example apps (incl. `multitenant/`, `native-auth-app/`, `saas/`, `streaming/`) were not covered, yet they are the patterns users copy into production. An example with disabled auth, permissive CORS, or a hardcoded JWT secret is an effective security finding the audit could not produce.

5. **Runtime behavior.** The entire audit was static. The repo's dominant escaped-bug class is "compiles clean, panics/misbehaves at boot or under load" (axum route-literal incident, env-var overrides, lazy-init). No lens booted the server, ran release-smoke-style checks, or fuzzed the parser (`fuzz.yml` exists but was not part of the audit).

6. **Known coverage-debt artifacts ignored.** `ignored-tests.md` (199 lines of disabled tests) and the `mutants-s3-*` surviving-mutant reports are pre-computed maps of code where tests pass while behavior is broken; no lens cross-referenced them against the modules where the high findings landed. (Note: the S3 vacuous-test gap directly enabled H40.)

7. **Unlensed crates.** `fraiseql-test-support`, `fraiseql-test-utils`, `fraiseql-error`, and the umbrella `fraiseql` crate. test-support builds the DB/Redis env-URL harness all 16 Dagger integration suites depend on — a bug there (wrong default URL, swallowed connect error) produces false-green CI across the whole workspace, a meta-risk larger than most single findings.

8. **Docs/tutorial accuracy and the tutorial app.** `docs/`, `tutorial/` (ships its own Dockerfile/package.json/web JS app). Precedent: stale `compiled-schema-lifecycle.md` had wrong reload/version-guard claims caught only during #378. Misleading operator security guidance is an exploitable gap in practice, and the tutorial's JS dependency tree got no deps lens.
