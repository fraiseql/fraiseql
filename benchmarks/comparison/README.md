# FraiseQL Benchmark Comparison

Reproducible benchmark comparison between FraiseQL, Hasura, and PostGraphile against the same PostgreSQL database.

## Methodology

All three engines execute identical GraphQL queries against a shared PostgreSQL 16 instance with the same schema, data, and hardware. Results are collected using k6 with consistent test parameters.

### Environment

- **Database**: PostgreSQL 16, 4 vCPU, 16GB RAM, 1000 connections max
- **Test Data**: 100,000 users, 500,000 posts, 2,000,000 comments
- **Host**: 8 vCPU, 32GB RAM, Ubuntu 22.04
- **Network**: Loopback (client and server on same host)
- **Duration**: 5 minutes sustained load per test

### Engines Under Test

| Engine | Version | Configuration |
|--------|---------|--------------|
| FraiseQL | 2.0.0-beta.2 | Release build, connection pool size 20 |
| Hasura | Latest stable | Default configuration, connection pool size 20 |
| PostGraphile | Latest stable | Default configuration, connection pool size 20 |

## Test Queries

### Simple Query

```graphql
{ users(limit: 10) { id name email } }
```

### Nested Query

```graphql
{ users(limit: 5) { id name posts(limit: 3) { id title } } }
```

### Filtered Query

```graphql
{ users(where: { name: { eq: "test" } }) { id name email } }
```

## Running

### Prerequisites

```bash
# Install k6
brew install k6  # macOS
# or: sudo apt install k6  # Ubuntu

# Start all engines
docker compose up -d
```

### Execute Tests

```bash
# Run against FraiseQL
ENDPOINT=http://localhost:8080/graphql k6 run k6/simple-query.js \
  --out json=results/fraiseql-simple.json

# Run against Hasura
ENDPOINT=http://localhost:8081/v1/graphql k6 run k6/simple-query.js \
  --out json=results/hasura-simple.json

# Run against PostGraphile
ENDPOINT=http://localhost:8082/graphql k6 run k6/simple-query.js \
  --out json=results/postgraphile-simple.json
```

### Analyze Results

```bash
# Compare results
python3 scripts/compare.py results/
```

## Expected Results

Based on architectural differences:

### Where FraiseQL Wins

- **Deterministic queries**: Compiled SQL eliminates query planning overhead.
- **Simple field projections**: Compile-time column selection avoids runtime analysis.
- **High throughput sustained load**: No runtime interpretation overhead.

### Where FraiseQL Loses

- **Cold start**: Schema compilation adds setup time compared to runtime-interpreted engines.
- **Dynamic queries**: Ad-hoc filters still require some runtime SQL generation.
- **Ecosystem maturity**: Hasura has years of production hardening and a larger community.

### Honest Assessment

FraiseQL's compile-time approach provides measurably lower latency for known query patterns. For dynamic, ad-hoc GraphQL exploration, Hasura's runtime approach may be more flexible. Choose based on your workload characteristics.
